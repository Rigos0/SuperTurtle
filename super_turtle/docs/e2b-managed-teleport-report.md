# E2B Managed Teleport Report

Checked against live docs on March 13, 2026.

Product decision note: this report captures the E2B capability assessment and the cleaner long-term hosted posture. The currently chosen v1 onboarding flow is different in one respect: hosted Claude/Codex setup is initiated from the user's own machine during first managed teleport, reusing the local-browser bootstrap model already proven on Azure.

## Recommendation

Use E2B as the managed runtime for teleported turtles.

For production, the core stack should be:

- E2B Sandboxes as the per-user runtime
- One SuperTurtle custom E2B template for managed teleport workloads
- E2B pause/resume for idle lifecycle instead of long-lived always-on VMs
- E2B secure sandbox access and restricted public traffic by default
- Control-plane-owned sandbox registry keyed by `sandbox_id`, `template_id`, and user/account linkage
- Provider credentials stored in our control plane and injected into sandboxes ephemerally
- Claude Code authenticated with API-based auth, not browser OAuth, for managed sandboxes
- Codex authenticated with API keys, not ChatGPT interactive login, for managed sandboxes
- E2B PTY/command execution as the primary remote control path
- E2B SSH access kept only as an operator escape hatch

This is a better fit than continuing the current SSH-to-VM model. E2B already gives us custom images, resumable sandbox state, secure file transfer, interactive terminals, public/private port control, sandbox listing by metadata, and prebuilt agent templates for Claude Code and Codex.

## What E2B helps with

### 1. Managed runtime replacement for the current VM

E2B sandboxes are isolated Linux microVMs with filesystem, terminal, process state, and network support. The docs explicitly support pause/resume with preserved filesystem and memory state, and `Sandbox.connect()` can resume a paused sandbox by ID.

For us, this means a teleported turtle can become:

- one sandbox per paid account
- paused when idle
- resumed for `/teleport`, `/status`, or follow-up work
- tracked by sandbox ID instead of SSH hostname

This directly matches the current managed-instance contract better than a traditional VM if we are comfortable with sandbox lifecycle semantics instead of raw machine lifecycle semantics.

### 2. Custom templates

E2B Build System 2.0 is the right path. We should build one production template for SuperTurtle instead of relying on ad hoc bootstrap on first launch.

That template should install and pin:

- Ubuntu 24.04 base
- Node 24
- Bun
- Python 3.13
- git
- ripgrep
- tmux
- rsync if we still need it during the transition
- `@anthropic-ai/claude-code`
- `@openai/codex`
- SuperTurtle runtime dependencies

We should also bake in:

- startup scripts
- health checks
- log paths
- system directories for provider config
- optional MCP server pre-pulls if we standardize on a fixed set

### 3. Fast warm starts

Templates snapshot the prepared environment, and E2B says those snapshots can load in a few hundred milliseconds. That is exactly what we want for a managed teleport target: the environment should already contain the full toolchain before the handoff starts.

### 4. Secure operator and app access

E2B secure access is on by default in SDK v2+, and public traffic can be restricted with `allowPublicTraffic: false`. If we ever expose a web UI, VNC stream, health endpoint, or debug console from inside the sandbox, we should require the E2B traffic token and front it with our control plane.

### 5. Fleet control

E2B lets us attach metadata to sandboxes and list/filter sandboxes by metadata. We should use metadata for non-secret routing only:

- `user_id`
- `account_id`
- `sandbox_role=managed-teleport`
- `driver=claude|codex`
- `teleport_session_id`
- `environment=prod|staging`

Do not put secrets in metadata. The docs mention API keys as a possible metadata use case, but for our product that is the wrong security boundary because metadata is explicitly designed for later lookup/listing.

## Auth recommendations

### Claude Code

For managed sandboxes, use API-backed auth, not interactive browser OAuth.

Why:

- E2B’s Claude docs show `ANTHROPIC_API_KEY` passed as a sandbox env
- Claude Code settings support `ANTHROPIC_API_KEY`
- Claude Code also supports `apiKeyHelper`, which is useful if we want short-lived tokens generated by a local helper instead of static env vars
- browser login stores state under the normal Claude config path, which is poor for a multi-tenant hosted product

Recommended production shape:

- Store the user’s Claude credential in the control plane
- Encrypt it at rest
- Inject it only into the sandbox session that belongs to that user
- Prefer `apiKeyHelper` or per-session env injection over persisting long-lived secrets into the sandbox filesystem
- Use Claude Code managed settings or user settings to enforce safe defaults and secret-deny rules

Good uses of Claude configuration inside E2B:

- `/etc/claude-code/managed-settings.json` or user-scoped settings for policy
- `permissions.deny` to block accidental reads of `.env`, `secrets/`, and control-plane injected files
- `.mcp.json` environment-variable expansion for authenticated MCP endpoints

What not to use as the default hosted path:

- `claude` browser login inside the managed sandbox
- copying a user’s `~/.claude.json` around between machines
- shared Anthropic credentials across users

### Codex

For managed sandboxes, use API-key auth, not Sign in with ChatGPT.

Why:

- E2B’s Codex docs show `CODEX_API_KEY` passed as a sandbox env
- OpenAI’s help docs say enterprise sign-in support is still coming later and recommend traditional API key provisioning for now
- ChatGPT login creates local stored credentials, which is a poor fit for a server-run multi-user managed product

Recommended production shape:

- Store a user-scoped Codex/OpenAI credential in the control plane
- Encrypt it at rest
- Inject `CODEX_API_KEY` only into that user’s sandbox/session
- Treat ChatGPT login as optional only for user-managed sandboxes where the user is explicitly operating their own environment

What not to use as the default hosted path:

- `codex --login` in our managed fleet
- copying `~/.codex/auth.json` into hosted sandboxes
- one OpenAI org key shared across customers

## Best runtime model

Use one sandbox per paid account, with explicit state in our control plane:

- `requested`
- `provisioning`
- `running`
- `paused`
- `failed`
- `deleted`

Map these onto E2B lifecycle state rather than treating the sandbox like a generic VM:

- create sandbox from the SuperTurtle template
- store `sandbox_id`
- set a short active timeout
- use lifecycle timeout-to-pause
- resume through `Sandbox.connect()` when needed
- kill only for reprovision/delete

For v1, I recommend:

- `onTimeout: "pause"`
- no autonomous auto-resume for billing-critical flows
- control-plane-driven resume on `/teleport`, `cloud status`, or explicit resume

Reason: it keeps entitlement enforcement and audit logging in our control plane instead of letting the runtime wake itself without a product-level decision.

## What to change in teleport

The current managed teleport contract returns an SSH target. If we move to E2B, that should change.

The better v1 contract is:

- `/v1/cli/teleport/target` returns sandbox identity and launch metadata, not SSH host data
- bot control plane resolves or resumes the target sandbox
- handoff bundle is uploaded with E2B file APIs
- remote bootstrap is started with `sandbox.commands.run(...)` or PTY
- logs stream back through E2B command/PTY output

So the new teleport path becomes:

1. Resolve user entitlement and managed sandbox record.
2. Create or resume the sandbox.
3. Upload handoff bundle and any required repo artifacts.
4. Start SuperTurtle in the sandbox.
5. Mark teleport session active in the control plane.

This lets us remove SSH host management from the customer-facing path.

## Technology choices

### Use now

- E2B SDK for sandbox lifecycle, commands, PTY, files, metadata, and connect/resume
- E2B Build System 2.0 for the production template
- One custom `superturtle-teleport` template
- E2B pause/resume for idle behavior
- E2B secure access defaults
- Restricted public traffic for all exposed sandbox ports
- E2B PTY for interactive control

### Use optionally

- E2B Desktop template only for fallback credential setup, browser-required auth, or future computer-use flows
- E2B SSH access for operator debugging only
- MCP image pre-pull if we standardize on shared hosted MCP servers

### Avoid as the primary design

- Per-user raw GCP VM provisioning if E2B is now the runtime provider
- Browser OAuth stored directly inside managed sandboxes for Claude/Codex
- Secrets in E2B sandbox metadata
- Public unauthenticated exposed sandbox ports
- Direct customer dependence on SSH hostnames or SSH config

## Recommended product architecture

### Managed by us

- Control plane owns the E2B API key and sandbox lifecycle
- Each paid account has exactly one logical managed sandbox
- Provider credentials are user-scoped records in our DB
- Sandbox gets secrets only for that user and only at runtime
- Idle sandboxes pause automatically
- Resume is initiated by the control plane

### Managed by user

Support this as a second provider mode, not a different product architecture:

- `runtime_provider = e2b_managed | e2b_byok`

In BYOK mode:

- user supplies E2B credentials or links their E2B account
- user can choose interactive CLI auth inside their own sandbox if they want
- our control plane still uses the same sandbox contract and teleport flow

## Implementation order

1. Replace the managed-instance abstraction so it can represent an E2B sandbox, not only a VM.
2. Add a `sandbox_id`-based hosted target contract and stop returning SSH coordinates for E2B-backed accounts.
3. Build the `superturtle-teleport` E2B template.
4. Add a sandbox lifecycle adapter in `../superturtle-web` for create, connect/resume, pause, kill, list, and metadata lookup.
5. Add provider credential storage and runtime injection for Claude and Codex.
6. Update `/teleport` to upload the handoff bundle through E2B instead of SSH/rsync for E2B-backed targets.
7. Keep SSH/manual teleport as a fallback path during migration.

## Bottom line

E2B is a strong fit for managed teleport.

The production shape we should use is not "log into tools manually inside a sandbox." It is:

- custom E2B template
- one sandbox per account
- pause/resume lifecycle
- control-plane-managed resume
- ephemeral secret injection
- API-based Claude/Codex auth
- PTY/files/commands instead of SSH as the normal path

That gives us a cleaner hosted product than continuing to build VM orchestration ourselves.
